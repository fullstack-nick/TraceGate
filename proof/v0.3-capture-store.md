# v0.3 Capture Store Live Proof

Date: 2026-07-06

## Clean Release Marker

The first live proof run validated the feature behavior, but the image tag pointed at the pre-v0.3 base commit because the deploy script tagged from `HEAD` while building an uncommitted worktree. That caveat is resolved.

The currently deployed image was built from a clean committed worktree:

```text
3ca83e59c49144da37d4202edca32bb6f4018ece
```

The deployed Docker tag is:

```text
tracegate:3ca83e59c491
```

Relevant commits:

```text
3ca83e5 Guard GCP deploys against dirty worktrees
f78b538 Implement v0.3 capture store
```

`deployments/gcp/scripts/deploy.ps1` now refuses to deploy a dirty worktree with an implicit Git-SHA image tag unless an explicit `-ImageTag` or `-AllowDirty` override is passed.

## Result

v0.3 Capture Store passed the local and live gates from a clean release marker:

- The deployed image tag maps to committed SHA `3ca83e59c491`.
- SQLx SQLite storage initializes with WAL and migrations.
- Failed and slow requests persist.
- Sensitive query params and headers are absent from logs and storage.
- Large response bodies are bounded and marked truncated.
- Retention and backup commands work locally and on the VM.
- Capture metrics are exported, with `tracegate_capture_dropped_total 0`.
- The deploy script now prevents implicit dirty-worktree tags from recreating the original mismatch.
